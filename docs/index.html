01 / START

Install, then opt in.

StarRouter requires Node.js 22.19 or newer and Pi 0.80.6 or newer; releases 0.80.6–0.80.10 are explicitly validated. It ships as one Pi extension and starts disabled.

$ pi install npm:pi-star-router /reload /router settings /router on
Conservative default

Nothing switches until enabled.

Auto-accept is also off. The first routing decision presents a focused recommendation that can be confirmed or cancelled.

One-off evaluation: pi -e npm:pi-star-router

02 / FLOW

A route is evidence through gates.

Prompt semantics stay local. External benchmark data informs candidates, but provider, identity, capability, quality, and confidence rules can stop a switch.

No classifier model. No cross-provider fallback. No invented benchmark match.

03 / MATCH

Names are not identities.

Creator, generation, and reasoning identity stay strict. Host scope decides which metrics are trustworthy, not which model identity wins.

Candidate generation

  • Configured provider only
  • Pi-available text models
  • Applied family/model filters
  • Supported thinking variants only

Evidence compatibility

  • Creator and family/subfamily
  • Generation and fixed version
  • Moving-alias protection
  • Reasoning/non-reasoning parity
  • Host-verified or model-only scope

Aggregators may use model quality from another host, but AA host price, speed, latency, token burn, and context fallback are suppressed.

Final safeguards

  • Vision/context hard gates
  • Thinking floors and caps
  • Relative quality floor
  • Pareto frontier
  • Confidence abstention

Qualified host labels remain distinct; true duplicates select one coherent row rather than a synthetic best-of-fields row. Host-verified evidence requires an exact normalized host alias—never a model API ID—and outranks model-only. Specialization stays a soft score. Cheapest and fastest abstain when their primary evidence is absent. Unknown optional AA prompt buckets are ignored without widening the routing profile set. Confidence measures evidence quality; it is not a calibrated probability of task success. Read the algorithm note.

04 / CONTROL

One focus. Explicit commit.

Route selection, settings, and filters use drafts and clear commit/cancel semantics. There is no Space-to-mark state.

Route confirmation

↑ ↓ move focus
Enter confirm focused route
Esc cancel and keep current

Current and recommended routes are distinct actions. Recommendation basis and application origin remain independent, so a manual alternative never rewrites the kernel recommendation. Cancellation persists a clear marker so old ambient UI does not return on branch reload.

Runtime modes

TUI gets the responsive confirmation and themed compact widget. RPC uses Pi's serializable selection API and plain text widget lines. Print/headless prints status, refuses TUI-only settings, and keeps current when confirmation cannot be shown. Auto-accept must be enabled globally for intentional unattended switching.

05 / TRUST

A repository is not an administrator.

Global user configuration owns privileged behavior. Project configuration is deliberately projected onto a smaller safe schema.

Global-only authority

  • Enable/disable routing
  • Select routing provider
  • Enable auto-accept
  • Change data source or API-key environment name

~/.pi/agent/model-router.json

Project-safe overrides

  • Objective and numeric thresholds
  • Show advanced settings
  • Provider/family/model filters
  • AA-slug alias pins (identity gates remain)

.pi/model-router.json

Prompt text, images, conversations, provider responses, and credentials are absent from the public benchmark cache. Cache/config/preset writes are atomic; validated network data remains usable if cache persistence fails. Config and preset reads are bounded. Session state stores enabled state, compact decision summaries, and decision-clear markers—not raw prompts.

06 / PROOF

Evidence, not projections.

Release checks distinguish deterministic product behavior from mutable external catalog observations. No monthly total or universal savings percentage is claimed.

$ npm test # 126 offline tests; standard Node network entry points denied $ npm run golden # 80/80 checked-in prompt profiles $ npm run typecheck $ npm run smoke $ npm audit --audit-level=low $ npm pack --dry-run --json

Opt-in network observation

npm run test:live
# 8 checks
# 2 requests: one per external source/file

npm run test:all
# offline, then live

The live job runs manually or on a schedule, reports external drift as a failure, and is not a pull-request gate. Model availability and prices can change without a StarRouter code change.

How to validate price impact

Record the observation time, exact provider/model IDs, input/output/cache token counts, dynamic provider fees, retry/tool behavior, and baseline selection policy. Compare the same workload under the same catalog snapshot. Re-run when prices change. A cheaper model price does not by itself prove equal task quality or universal savings.

07 / LIMITS

Deterministic is not omniscient.

The design favors inspectable failure modes over silent cleverness.

Known limits

  • Lexical signals can miss novel phrasing.
  • Benchmarks may not predict a private workload.
  • AA schema, optional buckets, rows, and prices can drift.
  • Provider metadata can omit capabilities.
  • Alias pins can be wrong but cannot bypass identity.
  • The offline preload is a test guard, not an OS sandbox.

Built-in response

  • Ask by default.
  • Stay within one provider.
  • Reject incompatible identities.
  • Use validated stale data only.
  • Abstain when evidence is insufficient.
